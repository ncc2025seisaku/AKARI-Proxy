# 発表用負荷テストデータ（デモサーバー: 127.0.0.1, v2+Brotli応答）
- 実行コマンド: `uv run --python 3.11 python loadtest/disaster_suite.py --demo-server --no-event-log --host 127.0.0.1 --port 0`
- すべてローカルで実行。詳細ログ: `logs/disaster_suite_summary.json`, `logs/disaster_suite_history.jsonl`

## シナリオ別結果（成功/総リクエスト, timeout数, 平均応答時間）
- ベースライン: 80/80 成功, timeout 0, 平均0.007s
- 遅延3000ms+ロス20%: 163/200 成功, timeout 37, 平均0.147s
- ジッター350ms相当: 200/200 成功, timeout 0, 平均0.186s
- ロス25%: 178/220 成功, timeout 42, 平均0.010s
- バースト高並列: 765/800 成功, timeout 35, 平均0.019s
- 中負荷持続: 400/400 成功, timeout 0, 平均0.043s
- 断続ドロップ: 120/150 成功, timeout 30, 平均0.002s
- 強フラップ: 176/200 成功, timeout 24, 平均0.007s
- フラップ緩和（再送強化）: 174/200 成功, timeout 26, 平均0.006s
- MTU揺れ近似: 223/240 成功, timeout 17, 平均0.029s
- 巨大レスポンス10MB: Brotli圧縮・チャンク分割で 6/10 成功, timeout 4, 平均0.382s（大きめの応答は少し詰まりがち）
- 3000本相当ロード: 2932/3000 成功, timeout 68, 平均0.065s

## ひとこと
- v2は全応答をBrotli圧縮＋チャンク化して送信。通常サイズは高速、巨大レスポンスは再送チューニング余地あり。
- フラップ系はNACK/再送強化で8割強まで復帰。ハートビート間隔をさらに広げるとtimeoutをもう少し減らせそう。
- 遅延・ロス極端時も最低限応答を維持。超大容量だけは余裕を持ったタイムアウト設定推奨。
