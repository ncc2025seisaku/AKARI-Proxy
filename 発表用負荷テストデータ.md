# 発表用負荷テストデータ（デモサーバー: 127.0.0.1, v2+Brotli応答, 非同期デフォルト）
- 実行コマンド: `uv run --python 3.11 python loadtest/disaster_suite.py --demo-server --no-event-log --host 127.0.0.1 --port 0`
- すべてローカルで実行。詳細ログ: `logs/disaster_suite_summary.json`, `logs/disaster_suite_history.jsonl`

- 遅延3000ms+ロス20%: 163/200 成功, timeout 37, 平均0.149s
- ジッター350ms相当: 200/200 成功, timeout 0, 平均0.175s
- ロス25%: 170/220 成功, timeout 50, 平均0.003s
- バースト高並列: 761/800 成功, timeout 39, 平均0.009s
- 中負荷持続: 400/400 成功, timeout 0, 平均0.044s
- 断続ドロップ: 126/150 成功, timeout 24, 平均0.002s
- 強フラップ: 177/200 成功, timeout 23, 平均0.004s
- フラップ緩和（再送強化）: 177/200 成功, timeout 23, 平均0.004s
- MTU揺れ近似: 230/240 成功, timeout 10, 平均0.028s

- 3000本相当ロード: 2934/3000 成功, timeout 66, 平均0.038s

## ひとこと
- タイムアウトとNACK上限を少し広げ、先頭チャンクの複製と大容量パリティONで粘りを強化。とはいえ巨大レスポンスはまだ厳しく、再送間隔やタイムアウトをさらに伸ばす余地あり。
- NACK連打は抑制済み。非同期サーバーで同時処理を維持しつつ、`AKARI_HEAD_DUP_COUNT`や`AKARI_LARGE_BODY_THRESHOLD`で冗長度を環境に合わせて可変化。
- バースト/多並列は高速。成功数とtimeoutのバランスをチューニングしながら、災害時の「最低限動く」状態をキープする方向。
