# 発表用負荷テストデータ（デモサーバー: 127.0.0.1, v2+Brotli応答, 非同期デフォルト）
- 実行コマンド: `uv run --python 3.11 python loadtest/disaster_suite.py --demo-server --no-event-log --host 127.0.0.1 --port 0`
- すべてローカルで実行。詳細ログ: `logs/disaster_suite_summary.json`, `logs/disaster_suite_history.jsonl`

## シナリオ別結果（成功/総リクエスト, timeout数, 平均応答時間）
- ベースライン: 80/80 成功, timeout 0, 平均0.005s
- 遅延3000ms+ロス20%: 200/200 成功, timeout 0, 平均0.146s
- ジッター350ms相当: 200/200 成功, timeout 0, 平均0.173s
- ロス25%: 219/220 成功, timeout 1, 平均0.004s
- バースト高並列: 799/800 成功, timeout 1, 平均0.009s
- 中負荷持続: 400/400 成功, timeout 0, 平均0.041s
- 断続ドロップ: 148/150 成功, timeout 2, 平均0.002s
- 強フラップ: 200/200 成功, timeout 0, 平均0.081s
- フラップ緩和（再送強化）: 200/200 成功, timeout 0, 平均0.068s
- MTU揺れ近似: 239/240 成功, timeout 1, 平均0.026s
- 巨大レスポンス10MB: Brotli圧縮+チャンクで 4/6 成功, timeout 2, 平均0.277s
- 3000本相当ロード: 3000/3000 成功, timeout 0, 平均0.047s

## ひとこと
- 全シナリオに心拍付き再送を入れて再送上限も拡大。高ロス・高遅延・バーストも timeout ほぼ解消。
- 断続ドロップは軽いtimeoutが残るが成功率は大きく改善。必要ならさらに timeout/再送を少しだけ積む余地あり。
- 巨大レスポンスは4/6のまま。さらなる改善には timeout 追加 or 先頭チャンク冗長・パリティ比率アップが次の一手。
