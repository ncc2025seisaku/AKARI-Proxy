# 発表用負荷テストデータ（デモサーバー: 127.0.0.1, v2+Brotli応答, 非同期デフォルト）
- 実行コマンド: `uv run --python 3.11 python loadtest/disaster_suite.py --demo-server --no-event-log --host 127.0.0.1 --port 0`
- すべてローカルで実行。詳細ログ: `logs/disaster_suite_summary.json`, `logs/disaster_suite_history.jsonl`

## シナリオ別結果（成功/総リクエスト, timeout数, 平均応答時間）
- ベースライン: 80/80 成功, timeout 0, 平均0.005s
- 遅延3000ms+ロス20%: 164/200 成功, timeout 36, 平均0.149s
- ジッター350ms相当: 200/200 成功, timeout 0, 平均0.182s
- ロス25%: 156/220 成功, timeout 64, 平均0.007s
- バースト高並列: 754/800 成功, timeout 46, 平均0.009s
- 中負荷持続: 400/400 成功, timeout 0, 平均0.042s
- 断続ドロップ: 114/150 成功, timeout 36, 平均0.004s
- 強フラップ: 175/200 成功, timeout 25, 平均0.005s
- フラップ緩和（再送強化）: 176/200 成功, timeout 24, 平均0.006s
- MTU揺れ近似: 221/240 成功, timeout 19, 平均0.028s
- 巨大レスポンス10MB: Brotli圧縮+チャンクで 7/10 成功, timeout 3, 平均0.402s（超大容量はタイムアウト寄り）
- 3000本相当ロード: 2950/3000 成功, timeout 50, 平均0.039s

## ひとこと
- 非同期デモサーバーで同時処理を強化。フラップ・高遅延・高ロスでも大半が完走、timeoutは設定でまだ削れる余地あり。
- 巨大レスポンスは依然シビア。再送間隔や全体タイムアウトを少し広げると成功率を上げられる見込み。
- バースト/多並列は高速。成功数とtimeoutのバランスは環境に合わせてチューニング可能。
