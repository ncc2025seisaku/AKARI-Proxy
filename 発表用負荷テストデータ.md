# 発表用負荷テストデータ（デモサーバー: 127.0.0.1:14600, 応答gzip）
- 実行コマンド: `uv run --python 3.11 python loadtest/disaster_suite.py --demo-server --no-event-log --host 127.0.0.1 --port 14600 --compress-response gzip`
- すべてローカルで実行。詳細ログ: `logs/disaster_suite_summary.json`, `logs/disaster_suite_history.jsonl`

## シナリオ別結果（成功/総リクエスト, timeout数, 平均応答時間）
- ベースライン: 80/80 成功, timeout 0, 平均0.003s
- 遅延3000ms+ロス20%: 163/200 成功, timeout 37, 平均0.153s
- ジッター300ms: 200/200 成功, timeout 0, 平均0.173s
- ロス25%: 153/220 成功, timeout 67, 平均0.003s
- バースト高並列: 774/800 成功, timeout 26, 平均0.013s
- 中負荷持続: 400/400 成功, timeout 0, 平均0.044s
- 断続ドロップ: 124/150 成功, timeout 26, 平均0.002s
- 強フラップ: 173/200 成功, timeout 27, 平均0.005s
- フラップ緩和（再送強化）: 170/200 成功, timeout 30, 平均0.003s
- MTU揺れ近似: 233/240 成功, timeout 7, 平均0.027s
- 巨大レスポンス10MB: gzip圧縮で 10/10 成功, timeout 0, 圧縮後9,751B, チャンク9個, 平均0.016s
- 3000本相当ロード: 2935/3000 成功, timeout 65, 平均0.044s

## ひとこと
- 大きなレスポンスは gzip圧縮+チャンク分割で小分け配信し、短時間で完走。
- フラップ系は NACK強化で成功率を底上げ。さらに再送間隔/ハートビート調整で伸びしろあり。
- 極端な遅延・ロス条件でも最低限応答を維持。timeout分は再送間隔を少し広げると改善余地。
