# Issue優先度分析・選出ワークフロー

オープンなIssueを分析し、今着手すべきIssueを決定する手順です。

## 1. 現在のIssue一覧取得

```bash
# オープンIssue一覧
gh issue list --state open --json number,title

# CI状態も確認（ブロッカーの有無）
gh run list --limit 5
```

## 2. 依存関係の分析

以下の観点でIssue間の依存関係を整理：

### 依存パターン
| パターン | 説明 | 例 |
|---------|------|-----|
| A → B | AはBが完了してから | リリース → CI安定化 |
| A // B | 並列実行可能 | ドキュメント // テスト |
| A ⊂ B | AはBの一部 | README整備 ⊂ リリース準備 |

### 典型的な依存チェーン
```
リリース準備
    ├── CI安定化 ← 最優先
    │       └── CIビルドテスト追加
    ├── ドキュメント整備
    │       ├── Flutter README
    │       └── メインREADME
    └── CHANGELOG更新
```

## 3. 優先度スコアリング

各Issueを以下の基準で評価（各1-3点）：

| 基準 | 低(1) | 中(2) | 高(3) |
|------|-------|-------|-------|
| **ブロッカー度** | 他に影響なし | 1-2個が依存 | 3個以上が依存 |
| **緊急度** | いつでもOK | 近々必要 | 今すぐ必要 |
| **工数** | 2時間以上 | 1-2時間 | 30分以内 |
| **リスク** | 大規模変更 | 中規模 | 小規模/安全 |

**合計点が高いものから着手**

## 4. 選出基準チェックリスト

着手するIssueは以下を全て満たすべき：

- [ ] 依存するIssueが全て完了している
- [ ] CIが壊れていない（または修正Issue）
- [ ] 明確なゴールがある
- [ ] 今の作業時間で完了できる

## 5. 推奨着手順のテンプレート

一般的な優先順位：

1. **CI安定化** - 土台が壊れていると何もできない
2. **ブロッカーになっているバグ修正**
3. **ドキュメント整備** - 依存がなく安全に進められる
4. **機能改善** - 安定したら機能追加
5. **テスト追加** - 機能が固まってから
6. **リリース準備** - 全て揃ってから

## 6. 決定の記録

選出理由を明確にする：

```markdown
## 選出: #XX タイトル

**理由**:
- ブロッカー度: X点（他のIssue Y,Z が依存）
- 緊急度: X点（理由）
- 工数: X点（推定時間）
- リスク: X点（影響範囲）

**合計: XX点**
```

## Tips

- **迷ったら依存なしのものから** - 安全に進められる
- **大きいIssueは分割** - 1-2時間で終わる単位に
- **CIが赤なら最優先** - 他の作業の土台
